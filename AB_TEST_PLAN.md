# План A/B-тестирования моделей прогнозирования дефолта

## 1. Постановка эксперимента

| Параметр | Значение |
|---|---|
| **Контрольная группа (A)** | Модель v1 — LogisticRegression |
| **Тестовая группа (B)** | Модель v2 — GradientBoostingClassifier |
| **Разделение трафика** | 50% → v1, 50% → v2 (случайное, per-request) |
| **Длительность** | 4 недели (достаточно для накопления ~10 000 наблюдений) |
| **Единица рандомизации** | Заявка (request_id) — одна заявка всегда обрабатывается одной версией |
| **Гипотеза (H₁)** | v2 имеет значимо более высокий F1-score на реальных запросах |
| **Нулевая гипотеза (H₀)** | F1(v2) ≤ F1(v1) |

### Как реализовано в коде

Запрос с `"version": "ab"` получает случайную версию (50/50) через `random.choice`.
Nginx-балансировщик (round-robin) автоматически чередует сервисы v1/v2 без изменения
клиентского кода.

```bash
# A/B-режим через API
curl -X POST http://localhost/predict \
  -H "Content-Type: application/json" \
  -d '{"LIMIT_BAL": 50000, ..., "version": "ab"}'

# Прямой запрос к v2 (для ручного сравнения)
curl -X POST http://localhost:5002/predict \
  -H "Content-Type: application/json" \
  -d '{"LIMIT_BAL": 50000, ...}'
```

## 2. Метрики оценки

### 2.1 Первичная метрика: F1-score (класс дефолта)

**Почему F1?** Датасет несбалансирован (~22% дефолтов). Accuracy вводит в
заблуждение — модель, предсказывающая «нет дефолта» для всех, получит accuracy 78%.
F1 балансирует Precision и Recall, что важно при ассиметричных потерях.

```
F1 = 2 × (Precision × Recall) / (Precision + Recall)
```

### 2.2 Вторичная метрика: Precision (по классу дефолта)

**Почему Precision?** Банк платит высокую цену за ложноположительные ошибки:
отказ добросовестному клиенту → потеря выручки + репутационный риск.
Высокий Precision означает, что из всех «предсказанных дефолтов» большинство
действительно дефолтные — меньше ошибочных отказов.

```
Precision = TP / (TP + FP)
```

### 2.3 Бизнес-метрики (см. ARCHITECTURE.md §8)

| Метрика | Описание |
|---|---|
| **ECL (Expected Credit Loss)** | Суммарные ожидаемые потери портфеля |
| **Approval Rate @ Precision≥0.85** | Доля одобренных заявок при фиксированном уровне точности |

## 3. Статистический анализ

### 3.1 Критерий: двусторонний z-тест для долей

F1-score не является долей, поэтому сравниваем **долю правильно классифицированных**
дефолтных клиентов (Recall) как прокси. Для сравнения двух долей:

```
H₀: p_v1 = p_v2
H₁: p_v1 ≠ p_v2

z = (p̂_v2 - p̂_v1) / sqrt(p̂(1-p̂)(1/n₁ + 1/n₂))

где p̂ = (x₁ + x₂) / (n₁ + n₂) — пулированная доля
```

При n₁ = n₂ ≥ 5 000 тест имеет мощность 80% для обнаружения разницы ≥ 0.02.

### 3.2 Доверительные интервалы (95%)

```python
import numpy as np
from scipy import stats

def wilson_ci(successes, n, alpha=0.05):
    """Интервал Вилсона — устойчив при малых выборках."""
    z = stats.norm.ppf(1 - alpha / 2)
    p = successes / n
    center = (p + z**2 / (2*n)) / (1 + z**2 / n)
    margin = z * np.sqrt(p*(1-p)/n + z**2/(4*n**2)) / (1 + z**2/n)
    return center - margin, center + margin
```

Если ДИ для v2 не пересекается с ДИ для v1 — разница статистически значима.

### 3.3 Критерий успешности

| Условие | Решение |
|---|---|
| F1(v2) > F1(v1) И p-value < 0.05 | ✅ Перевести 100% трафика на v2 |
| F1(v2) ≤ F1(v1) ИЛИ p-value ≥ 0.05 | ❌ Оставить v1, не внедрять v2 |
| Значимое снижение Precision у v2 | ⚠️ Дополнительный анализ, перенастройка порога |

**Минимальный практически значимый эффект (MDE):** ΔF1 ≥ 0.02
(менее 2% улучшения F1 не окупает сложность перехода).

## 4. Влияние A/B-теста на архитектуру сервиса

```
                     ┌──────────────────────┐
  Входящий трафик    │      Nginx            │
  ─────────────────► │  round-robin 50/50   │
                     └───────┬──────┬───────┘
                             │      │
                    ┌────────▼─┐  ┌─▼────────┐
                    │ v1 :5001 │  │ v2 :5002 │
                    └────┬─────┘  └─────┬────┘
                         │              │
                    ┌────▼──────────────▼────┐
                    │  Логи (JSON stdout)     │
                    │  → ELK / ClickHouse     │
                    │  поле "model_version"   │
                    │  позволяет разделить    │
                    │  метрики v1 и v2        │
                    └────────────────────────┘
```

Каждая строка лога содержит `"model_version": "v1"` или `"v2"`,
что позволяет в любой момент извлечь метрики по каждой группе.

### Пример агрегации логов для анализа

```python
import pandas as pd

logs = pd.read_json("predictions.jsonl", lines=True)

for version, group in logs.groupby("model_version"):
    # Для анализа нужны реальные метки (поступают через несколько недель)
    print(f"\nВерсия {version}: {len(group)} запросов")
    print(f"  Средняя вероятность дефолта: {group['probability'].mean():.3f}")
    print(f"  Доля предсказанных дефолтов: {group['prediction'].mean():.3f}")
```
