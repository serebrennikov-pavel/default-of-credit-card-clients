# Архитектура сервиса прогнозирования дефолта

## 1. Монолит vs Микросервисы

### Почему выбран монолит для данного проекта

В рамках учебного проекта выбран **монолитный подход**: Flask-приложение,
Gunicorn-воркеры, один Docker-образ.

| Критерий | Монолит (выбран) | Микросервисы |
|---|---|---|
| Сложность развёртывания | ✅ Низкая | ❌ Высокая (k8s, service mesh) |
| Время до первого деплоя | ✅ Часы | ❌ Дни/недели |
| Масштабирование | ❌ Горизонтальное только целиком | ✅ Независимо по компонентам |
| Отказоустойчивость | ❌ Один упал — всё упало | ✅ Изолированные сбои |
| Подходит для нагрузки | ✅ До ~1 000 RPS | ✅ Неограниченно |

**Вывод:** для одной модели с предсказуемой нагрузкой (банковский скоринг обычно
до нескольких сотен запросов в минуту) монолит достаточен и проще в обслуживании.
Переход к микросервисам оправдан при появлении независимых компонент:
отдельного сервиса предобработки данных, сервиса мониторинга дрейфа,
пайплайна дообучения.

## 3. Брокер сообщений (концепт: RabbitMQ)

В гипотетическом сценарии масштабирования RabbitMQ решает две задачи:

### 3.1 Асинхронная батч-обработка

Банк еженощно прогоняет 500 000 клиентов через скоринг.
Синхронный HTTP-вызов заблокировал бы воркеры на часы.

```
┌──────────────┐   JSON сообщение   ┌─────────────┐   ┌──────────────────┐
│  Upstream    │ ─────────────────► │  RabbitMQ   │ ─►│  ML-воркеры (N)  │
│  система     │                    │  очередь    │   │  predict & store  │
└──────────────┘                    └─────────────┘   └──────────────────┘
```

Сообщение в очереди:
```json
{"client_id": "C-12345", "features": {...}, "requested_at": "2025-01-01T10:00:00"}
```

### 3.2 Логирование предсказаний для аудита

Каждое предсказание публикуется в отдельную очередь `predictions.log`,
из которой потребители пишут в аналитическое хранилище (ClickHouse, BigQuery).

## 4. Логирование и мониторинг

### Формат логов (JSON)

```json
{
  "time": "2025-01-01T10:00:05",
  "event": "prediction",
  "version": "v1",
  "prediction": 1,
  "probability": 0.7823,
  "duration_ms": 12.5
}
```

Логи пишутся в `stdout` контейнера. В production-среде ELK-стек (Elasticsearch +
Logstash + Kibana) собирает их через `docker log driver` и строит дашборды:
- доля предсказанных дефолтов по времени
- latency p50/p95/p99
- сравнение распределений v1 vs v2 для A/B-теста

## 5. MLOps инструменты

### DVC (Data Version Control)

**Назначение:** версионирование данных и ML-пайплайна.

В контексте проекта DVC позволяет:
- зафиксировать версию датасета UCI (хэш файла в `.dvc`)
- описать пайплайн как граф зависимостей:
  ```
  download_data → dataset.py → features.py → train.py → model_v1.joblib
  ```
- воспроизвести любой исторический эксперимент командой `dvc repro`
- хранить данные и модели в удалённом хранилище (S3, GCS), не засоряя Git

### MLflow

**Назначение:** отслеживание экспериментов и реестр моделей.

В контексте проекта MLflow позволяет:
- логировать параметры обучения (C=0.1, n_estimators=200) и метрики (F1, ROC-AUC)
  командой `mlflow.log_param()` / `mlflow.log_metric()`
- сравнивать run-ы на UI: понять, при каких гиперпараметрах v2 выиграла у v1
- зарегистрировать лучшую модель в Model Registry с тегами
  `staging` → `production`
- обслуживать модель через встроенный `mlflow models serve`

## 6. Конвертация в ONNX-ML

ONNX (Open Neural Network Exchange) — открытый формат для переносимости моделей.

**Зачем:** ONNX Runtime работает быстрее sklearn на CPU (особенно при батч-инференсе)
и позволяет деплоить модель без зависимости от Python/sklearn.

**Пример конвертации LogisticRegression:**

```python
from skl2onnx import convert_sklearn
from skl2onnx.common.data_types import FloatTensorType
import onnxruntime as rt

# Конвертируем Pipeline в ONNX
initial_type = [("float_input", FloatTensorType([None, n_features]))]
onnx_model = convert_sklearn(pipeline, initial_types=initial_type)

with open("models/model_v1.onnx", "wb") as f:
    f.write(onnx_model.SerializeToString())

# Инференс через ONNX Runtime
sess = rt.InferenceSession("models/model_v1.onnx")
pred = sess.run(["output_label"], {"float_input": X_test.astype("float32")})
```

**Ускорение:** на типичных 23 признаках — 2-3× быстрее sklearn predict().

## 7. uWSGI + NGINX в production

| Компонент | Роль |
|---|---|
| **NGINX** | Принимает HTTP-соединения, терминирует SSL, кэширует статику, балансирует нагрузку между воркерами, защищает от медленных клиентов |
| **uWSGI** | WSGI-сервер: держит пул Python-процессов, передаёт запросы в Flask через бинарный протокол uwsgi |
| **Flask** | Обрабатывает бизнес-логику предсказания |

```
Internet → NGINX (:443/80) → uWSGI socket → Flask app
```

uWSGI + NGINX не обязательны для учебного проекта — здесь используется Gunicorn,
который проще в настройке и достаточен для нагрузки до ~500 RPS.

## 8. Бизнес-метрики

Помимо технических метрик модели (F1, ROC-AUC), бизнес заинтересован в:

### 8.1 Ожидаемые финансовые потери (Expected Credit Loss, ECL)

```
ECL = Σ (probability_i × LGD × EAD_i)
```
где:
- `probability_i` — вероятность дефолта клиента i (выход модели)
- `LGD` (Loss Given Default) — доля потерь при дефолте, например 0.6
- `EAD_i` (Exposure at Default) — сумма задолженности клиента i

**Как считать:** умножить `probability` из ответа `/predict` на задолженность клиента.
Сравнить ECL модели v2 с v1 — если v2 точнее идентифицирует рисковых клиентов,
суммарный ECL портфеля снизится.

### 8.2 Доля одобрений при фиксированном уровне риска

```
approval_rate@threshold = sum(probability_i < threshold) / total_clients
```

При одном и том же пороге риска (например, 0.3) новая модель v2 может одобрить
больше клиентов, не увеличивая долю дефолтов — это прямой прирост выручки банка.

**Как считать:** на тестовой выборке сравнить `approval_rate` v1 vs v2 при
threshold, при котором precision ≥ 0.85.
